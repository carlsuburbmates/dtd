# Operations & troubleshooting

The system is autonomous; this document explains how to read it and how to recover when a loop misbehaves.

## 0. Non-technical operator routine (default)

This routine is for an operator with no coding background. Use `/ops` as the primary control view.

### Ownership and response times

1. Primary operator owner: solo founder/owner (`carlg`).
2. Response-time standard:
- `Monitor`: note same day.
- `Investigate`: start within 4 hours.
- `Escalate`: immediate in the same check window.

### Operator boundary rule (hard rule)

Use this rule for every action surfaced in `/ops` or linked operator workflow:

1. If an action can change locked runtime policy, it is always `technical-owner mode`.
2. If an action is reversible, bounded, and policy-safe, it can be `operator mode`.

Interpretation:
- `Operator mode`: safe day-to-day operations, bounded remediation, and read-only oversight.
- `Technical-owner mode`: infra/runtime intervention, policy changes, or any action that can alter locked decisions in `docs/governance/LOCK_STATE.md`.

### Daily routine

1. Open `/ops` at start of day.
2. Check these cards first:
- `Revenue Â· at risk`
- `Loop cards`
- `Alerts`
- `Discovery pending`
3. At midday and end of day, refresh `/ops` and re-check the same cards.
4. Log one short note in `/ops`: what changed, what action was taken, and whether follow-up is needed.

### Weekly routine

1. Review 7-day trends for intros, conversions, suppressed intros, and at-risk revenue.
2. Confirm no loop has been stale repeatedly.
3. Confirm unresolved high-severity alerts are zero.
4. Confirm priority remediation items have an owner and due date.

### Action options (always choose one)

1. `Monitor`: data is stable and within expected range.
2. `Investigate`: unusual movement appears; collect context and re-check within the same day.
3. `Escalate`: clear risk to billing, matching continuity, or trust signals; notify technical owner immediately.

### Escalation triggers (plain-language)

Escalate immediately if any of these occur:
1. Any core loop appears stopped or stale beyond 2x interval.
2. High-severity alert persists after refresh.
3. At-risk revenue rises for two consecutive checks.
4. Discovery backlog rises continuously without recovery.
5. Owner/trainer core paths appear broken from user reports.

## 3. Common failure scenarios

### A. A loop has stopped (last_run is far in the past)

Likely cause: backend crash or a long-running coroutine. Action:
1. `tail -n 200 /var/log/supervisor/backend.err.log` â€” look for tracebacks.
2. `sudo supervisorctl restart backend` â€” loops re-schedule on startup.
3. Check `/api/oversight` after 60 s â€” `last_run` should refresh.

### B. `intro_drop` alert (â‰Ą50 % drop in 24 h)

If the drop is real (not a side effect of yesterday's burst):
1. Check `audit_recent` for any high-impact changes (deletions, manual config writes).
2. Inspect `pricing_state` â€” confirm `pricing_mode=fixed` and `intro_fee_cents` equals your launch policy.
3. If a recent change is the cause, write its inverse as a fresh `config_snapshots` row; the next health loop will treat the previous change as outdated.

### C. `auto_rollback` alert

Means the health loop reverted a `config_snapshots` row because conversions cliffed. Expected next steps:
1. Read the rolled-back snapshot's `kind` and `payload`.
2. Either re-apply with adjusted parameters, or leave reverted (the system kept itself stable in the meantime).

### D. `fraud_suppressed` alert spikes

Often a single bad actor. Inspect:
```bash
mongo $MONGO_URL/$DB_NAME --eval 'db.intros.aggregate([
  {$match:{billing_status:"suppressed"}},
  {$group:{_id:"$ip", n:{$sum:1}}},
  {$sort:{n:-1}}, {$limit:5}
])'
```
No action is required â€” these intros never billed and never affected ranking. If you want to permanently block, add the IP to a future `blocklist` collection (already supported by `evaluate_intro` if you extend it).

### E. Discovery queue not draining

Loop runs every 10 min. Check `system_state.discovery.last_run`. If stale, see (A). If fresh but pending count is high, the queue may be backlogged; heuristic verification still resolves items conservatively (many go to `discarded`).

### F. Source ingestion failures or suppressed sources

Use this when `source_ingestion.failed_sources` or `source_ingestion.suppressed_sources` rises:
1. Open `/api/oversight` and inspect `loops.source_ingestion` for `failed_sources`, `skipped_suppressed`, `suppressed_sources`, and `reason_codes`.
2. If suppression is active, inspect `source_ingestion_state` for `source_url`, `consecutive_failures`, `last_error_code`, and `suppressed_until`.
3. Confirm the upstream page still responds and still contains candidate links.
4. If the source recovered, wait for the suppression window to expire, then verify the next run clears `consecutive_failures` and updates `last_ok_at`.
5. Escalate if the same source re-enters suppression repeatedly across daily checks.

### G. Billing recovery states rising

Use this when `Revenue Â· at risk` rises or billing retry states accumulate:

| Signal | Meaning | Operator action |
|---|---|---|
| `payment_failed` / `uncollectible` | Collection failed and retry loop may re-attempt. | Investigate trend; monitor whether rows move to `retry_sent` or `paid`. |
| `retry_sent` | Retry was issued and is awaiting outcome. | Monitor next checks; no manual action unless it stalls. |
| `retry_failed` | Retry attempt did not restore collection. | Investigate trainer billing health and watch remaining retry budget. |
| `retry_exhausted` | Automatic retry budget is spent. | Escalate and route trainer through billing remediation/support. |
| `needs_remediation` | Missing consent/profile/integration blocker prevents collection. | Send trainer to `/trainer/billing` remediation path or support email. |

Retry policy defaults:
- Maximum attempts: `3`
- Base backoff: `24h`
- Backoff doubles per attempt up to `168h`

Lifecycle outcome guide:
- `invoice_sent`: monitor for later settlement; no immediate action.
- `paid`: collected successfully; no action.
- `payment_failed`: watch retry progression and trainer billing health.
- `uncollectible`: escalate and move trainer to remediation/support path.
- `disputed`: escalate immediately and track until resolved.
- `dispute_resolved`: monitor; revenue is no longer at risk for that row.
- `waived`: intentional non-collection; monitor only.
- `refunded`: confirm reason and monitor for recurrence.
- `trial_free`: expected non-collection during free-intro window.
- `profile_incomplete` / `consent_required` / `stripe_unconfigured` / `invoice_error`: remediation path required before collection can resume.

### H. Pricing mismatch

If intro fee differs from expected launch policy, verify runtime env and restart runtime:
- `FIXED_INTRO_FEE_CENTS` should match your intended amount (default `500`).
- `TRAINER_FREE_INTRO_DAYS` should match your intended trial window (default `30`).

### I. Reactivation return-to-active is weak

Use `Reactivation summary` in `/ops`.

Thresholds:
- `Monitor`: fewer than 5 resolved cases in 7 days, or return-to-active rate is at least 50%.
- `Investigate`: 5 or more resolved cases in 7 days and `active_after_resolution_7d / resolved_7d` falls below 50%.
- `Escalate`: return-to-active rate falls below 25% across two consecutive daily checks, or open candidates keep rising while resolved remains flat.

Interpretation:
- `resolved_7d`: candidates the system believes exited the reactivation cohort.
- `active_after_resolution_7d`: resolved candidates that are back to published and above active confidence threshold.
- Return-to-active rate = `active_after_resolution_7d / resolved_7d` when `resolved_7d > 0`.

## 4. Manual interventions (when essential)

The system is designed so that none of the routine workflows need a human. Direct DB intervention is reserved for:
- Legal takedown of a real listing â†’ `db.trainers.updateOne({id:"â€¦"}, {$set:{published:false, verification_status:"hold"}})`.
- Refunding a billed intro/conversion â†’ flip `billing_status` to `refunded` (any non-`billed` value is excluded from revenue & ranking automatically).
- Wiping a poisoned discovery batch â†’ `db.discovery_queue.deleteMany({source:"â€¦"})`.

Every direct write should be paired with an `audit_log` insert noting the human actor and reason; this preserves the integrity of the snapshot timeline.
